# Mineradio 打包架构总结

本程序的打包架构是一套基于 **`electron-builder`** 并经过深度定制的 **NSIS** (Windows 安装包) 构建体系。它放弃了常规的压缩打包，转而追求运行期的极致灵活性，同时通过底层拦截确保了用户硬盘的文件安全。

整体的打包流和设计亮点包含以下五个核心维度：

## 1. 核心构建引擎 (electron-builder)
程序依赖 `package.json` 中的 `"build"` 配置块驱动 `electron-builder`。
- **构建目标**：`win32 (x64)`。
- **打包格式**：传统的 `NSIS` 格式，最终生成 `Mineradio-1.x.x-Setup.exe` 安装文件。

## 2. 源码散列化策略 (asar: false)
绝大多数标准的 Electron 项目都会将代码加密压缩成一个庞大的 `app.asar` 文件。但 Mineradio 在配置中明确将 `"asar"` 设置为了 `false`。
- **核心优势**：解包后的安装目录会保留完整的 `resources/app` 散装目录结构。这极大地便利了本地 Node 环境（如 `server.js`）在运行时的直接文件读写（例如加载本地 vendor 依赖、执行音频分析的子进程）。
- **热更新铺垫**：允许后续通过小体量的网络请求，直接替换远端的 `.json` 配置或单独的 `.js` 逻辑模块，从而实现免重装的“热更新”和静默补丁分发。

## 3. 底层安装流接管 (Custom NSIS 脚本)
项目并没有使用标准的默认安装向导，而是通过 `"include": "build/installer.nsh"` 注入了高度定制的 NSIS 安装宏来彻底接管安装流：
- **智能避开 C 盘**：脚本会在安装前自动检测硬盘，优先寻找电脑中的 D 到 Z 盘作为默认安装路径，仅在完全没有其他分区时才回退至 C 盘。
- **强制专用目录**：强制将应用安装在名为 `Mineradio` 的独立顶级文件夹下。
- **禁止路径修改**：设定了 `"allowToChangeInstallationDirectory": false` 并且关闭了一键安装（`"oneClick": false`），以实现对文件落地路径的绝对管控。这能彻底杜绝因用户误操作将程序安装到盘符根目录，而在后续触发误删全盘的严重后果。

## 4. 关键生命周期钩子 (afterPack)
配置了 `"afterPack": "build/after-pack.js"` 钩子函数。
- 在 `electron-builder` 刚完成源码复制、但尚未压缩成最终的 `.exe` 之前，这段 Node 脚本会及时介入。
- **安全拦截**：专门用于清理多余的系统构建残留，或是往生成的应用目录中注入关键的安全标识（如防爆破标记），确保新版卸载器可以准确无误地识别 Mineradio 专属文件体系。

## 5. 增量更新与分发基石
每次执行 `npm run build:win` 打包后：
- 不仅会生成 Setup.exe，还会同步输出对应的 `.blockmap` 和 `latest.yml` 文件。
- 这是 Electron 原生自动更新模块（`electron-updater`）运行的基础，让后续可以通过对比 blockmap 映射文件来实现极低带宽消耗的增量二进制分发。

---
> **总结**：这是一套 **“以牺牲单体文件封装为代价换取全盘运行自由度”**，并且 **“重度魔改 NSIS 安装层以防御用户误操作和环境灾难”** 的生产级桌面端打包方案。
